# AWS Well-Architected Review — Remediation

**Terraform, runbooks, and scripts that close all 27 findings from the AWS Well-Architected Framework review of a PMI ARAS Innovator PLM workload — 9 HIGH, 14 MEDIUM, 4 LOW across Security, Reliability, Performance, and Cost. Delivers $3,675/month ($44,100/year) in addressable cost savings while raising every pillar score by 26–34 points. Each finding's fix is traceable from the inventory YAML to a specific Terraform module, runbook, or script.**

## The single source of truth

[`findings/findings.yaml`](findings/findings.yaml) — 27 entries, one per WAF
finding. Each entry carries pillar, risk, status, the artefact that fixed it
(Terraform module, runbook, or script path), and the measured before/after
state. CI validates the schema on every PR and re-generates
[`docs/remediation-summary.md`](docs/remediation-summary.md) from it.

```yaml
- id: SEC-02
  pillar: security
  title: RDS SQL Server storage not encrypted
  risk: HIGH
  status: remediated
  fixed_by:
    module: terraform/modules/rds-mssql
    runbook: runbooks/rds-encryption-cutover.md
  before: "storage_encrypted = false; 2 TB of PLM data in plaintext"
  after:  "storage_encrypted = true with dedicated KMS CMK; rotation enabled"
  effort_days: 15
```

Adding or updating a finding is a one-file edit — the report and the CI
gate both pick it up automatically.

## Key decisions

A handful of non-obvious calls were made early — these shaped the rest of the remediation rollout.

- **The findings inventory is the source of truth, not the report.** The original `.docx` was the snapshot of *what we found*; for *what we did about it*, hand-editing a Word document over six sprints would have drifted within a week. The YAML inventory + auto-generated markdown means PR review of a finding update is a single-file diff that everyone can read in GitHub. CI rejects any PR where the YAML and the markdown disagree.

- **One module closes multiple findings when they share a resource.** SEC-02 (encryption), REL-01 (Multi-AZ), PERF-01 (rightsize), COST-02 (cost reduction), and PERF-07 (Query Store) are five findings on the same RDS instance. The [`rds-mssql`](terraform/modules/rds-mssql/) module sets the target state for all five and the runbook [`rds-encryption-cutover.md`](runbooks/rds-encryption-cutover.md) sequences them into a single 4-hour maintenance window. Treating them as five independent changes would have meant five maintenance windows and five rollback windows.

- **WAF deployed in COUNT mode, then promoted to BLOCK.** Switching a managed rule group straight to BLOCK in front of a complex application like ARAS guarantees a false-positive incident on day one. The [`waf`](terraform/modules/waf/) module takes a `rule_action` variable — Terraform applies COUNT for two weeks, the operator reviews logs against legitimate ARAS traffic, then flips the variable to BLOCK in a follow-up PR. Same Terraform, different setting, no surprise outage.

- **IAM least-privilege generated from CloudTrail observation, not hand-written.** SEC-01 replaced `PowerUserAccess` with a custom policy. Writing that policy by hand from the architecture diagram would have missed at least one API call the application actually uses, requiring an emergency rollback. Instead, [`scripts/iam-policy-generator.py`](scripts/iam-policy-generator.py) scans 30 days of CloudTrail events for the role and emits a draft policy — the human work is *reviewing and tightening* that draft (replacing `*` with specific ARNs, removing read-only enumeration that production doesn't need) rather than starting from a blank file.

- **Compute Savings Plan before RDS RI.** COST-01 was the highest cost-saving finding but had a sequencing dependency: PERF-01 right-sized RDS from `r5.2xlarge` to `r5.xlarge`, and PERF-04 changed the web tier from `t3.large` to `m5.large`. Buying RIs *before* those changes would lock in the wrong instance type for a year. The runbook [`savings-plan-purchase.md`](runbooks/savings-plan-purchase.md) orders the actions: Compute Savings Plan first (instance-family agnostic, safe to buy immediately), then RDS RI four weeks later once the rightsize has been validated. Sequencing alone saved ~$500/month that an unsequenced buy would have wasted.

- **PERF-06 is the one finding accepted with residual risk.** App-tier instances stayed on 1-minute CloudWatch monitoring (cost ~$2/instance/month) because the perf-investigation use case is real; web-tier moved to 5-minute baseline (COST-05). Documenting the trade-off explicitly is more honest than pretending every finding is "fully closed".

## How to use this repo

### Update a finding's status

1. Edit [`findings/findings.yaml`](findings/findings.yaml).
2. `python3 scripts/validate_findings.py` — confirm schema still passes.
3. `python3 scripts/generate_report.py` — regenerate the markdown summary.
4. Commit both files. CI fails any PR where the markdown is out of sync.

### Run the Terraform

```bash
cd terraform/environments/prod
terraform init
terraform plan -out=prod.tfplan
terraform apply prod.tfplan
```

The module count drives the apply order — most fixes can land independently.
Sequencing for the RDS cutover (SEC-02 + REL-01 + PERF-01 + PERF-07) is
documented in [`runbooks/rds-encryption-cutover.md`](runbooks/rds-encryption-cutover.md).

### Promote WAF from COUNT to BLOCK

Two weeks after the WAF module is deployed in `count` mode:

```bash
# Edit terraform.tfvars
sed -i 's/enable_waf_block_mode = false/enable_waf_block_mode = true/' \
    terraform/environments/prod/terraform.tfvars
terraform -chdir=terraform/environments/prod apply
```

### Quarterly Game Day

```bash
# Run the first scenario from runbooks/game-day.md
TEMPLATE_ID=$(terraform -chdir=terraform/environments/prod \
    output -json fis_experiment_template_ids | jq -r .terminate_ec2)
aws fis start-experiment --experiment-template-id "$TEMPLATE_ID"
```
